{# ─── What Is Private Inference ───────────────── #}

Overview

What is Private Inference?

Private inference means running AI models on hardware you control — your servers, your VPS, or your edge devices — instead of sending data to third-party API providers. The models are the same frontier and open-weight models you would use via Anthropic, OpenAI, or Google, but inference happens locally.

This is not about training models. It is about serving them. You download a pre-trained model, deploy an inference engine on your infrastructure, and make predictions via a local API — just like calling OpenAI, but every request stays inside your network boundary.

{# ─── API vs Self-Hosted ──────────────────── #}

Comparison

API vs Self-Hosted Inference

Most teams should start with API-based inference (Claude, GPT, Gemini) — it is the fastest path to value. Self-hosting makes sense when volume, privacy, or compliance requirements outweigh the operational overhead.

API-Based Self-Hosted
Time to value Minutes — sign up and call an API Days — provision hardware, download model, configure engine
Cost at low volume Pay per token — no fixed cost Fixed GPU/server cost regardless of usage
Cost at high volume Per-token cost grows linearly with usage Fixed cost — marginal cost per request near zero
Data privacy Data sent to third-party servers Data stays on your hardware
Compliance Depends on provider's certifications Full control — meet any regulatory requirement
Model selection Access to frontier models (Claude 4, GPT-5, Gemini 2.5) Limited to open-weight models
Latency Adds network round-trip time Sub-millisecond local API calls
Offline capable Requires internet connectivity Runs fully offline
Operational overhead Zero — provider manages infrastructure You manage hardware, model updates, monitoring

Our recommendation: Start with API-based inference to validate your use case and prove ROI. Migrate to self-hosted when your volume justifies the fixed infrastructure cost, your data privacy requirements demand it, or you need offline capability. We help you plan the transition at the right time.

Technology

Inference Engines We Recommend

We recommend open-source Rust-based inference engines — they produce small, fast binaries with zero external dependencies, ideal for production deployment on modest hardware.

General purpose

Candle

Minimalist ML framework from Hugging Face. Ideal for most self-hosted inference scenarios — small binary (~8 MB), broad model support (Llama, Mistral, Phi, Gemma, Stable Diffusion, Whisper, YOLO), GPU acceleration via CUDA and Metal, and first-class WASM support for browser or edge deployment.

  • • 20k+ GitHub stars, 260+ contributors — large, active community
  • • Quantization support via GGUF — run models on consumer GPUs
  • • Apache 2.0 licensed — fully open source
View on GitHub

Large models, single GPU

xinfer

Blazing-fast LLM inference in pure Rust with zero Python dependencies. Designed for running 30B+ parameter models on single consumer GPUs. Achieves up to 197 tok/s decode for large models with aggressive KV cache compression (TurboQuant 2-4 bit) and continuous batching.

  • • Flash attention, FlashInfer, CUDA Graphs, prefix caching
  • • Runs on older GPUs (V100) with NVFP4 support
  • • OpenAI and Anthropic-compatible API — drop-in replacement
  • • Built-in Web UI and MCP tool calling
View on GitHub

Which one should you use? Start with Candle for most workloads — it supports the widest range of models and use cases (text, image, audio, vision) with a tiny footprint. Use xinfer when you need to run large language models (30B+ params) on a single GPU and need maximum throughput per watt. Both produce single-binary deployments with zero Python or Node dependencies. We help you evaluate, choose, and deploy the right engine for your specific workload.

Use cases

When Private Inference Makes Sense

These are the scenarios where self-hosting inference delivers clear advantages over API-based approaches.

📄

Document Processing & OCR

Extract data from invoices, contracts, forms, applications, receipts, and business records using OCR automation and document AI. Private inference ensures document contents never leave your network — critical for banking, legal, and healthcare compliance.

🔒

Internal Knowledge Assistants

RAG over proprietary documents, internal wikis, and confidential data. Self-hosted inference keeps both the document corpus and the queries inside your infrastructure — no proprietary data ever reaches a third-party API.

🏛️

Regulated Chatbots

Customer-facing AI in regulated industries (banking, insurance, healthcare) where data residency laws prohibit sending customer data to foreign API providers. Vietnam AI Law compliance is a common driver.

⚡

Batch Processing

High-volume classification, summarisation, or extraction workloads where per-token API costs accumulate quickly. At scale, self-hosted inference provides predictable fixed costs and near-zero marginal cost per request.

📡

Offline & Edge Environments

Manufacturing plants, remote sites, ships, or military installations with limited or no internet connectivity. Candle compiles to WASM and runs in the browser — inference without any network dependency whatsoever.

💰

Cost Control at Scale

Teams processing millions of requests per month often find self-hosted inference breaks even within 3-6 months vs API pricing. After break-even, every additional request is nearly free.

👁️

Computer Vision

Run vision models locally for defect detection, ID verification, traffic analysis, retail shelf monitoring, and workplace safety. Self-hosted inference keeps video feeds and images on your infrastructure — essential for manufacturing, logistics, and security applications.

FAQ

Common Questions

Is self-hosted inference as good as API-based models?
For many workloads, yes. Open-weight models (Llama 3, Mistral, Qwen, Gemma, Phi) are competitive with frontier models on specific tasks. The trade-off is that frontier models (Claude 4, GPT-5, Gemini 2.5) are only available via API. We help you evaluate which tasks need frontier models and which can be served by self-hosted open-weight models — most teams find a hybrid approach works best.
What hardware do I need?
It depends on the model size and throughput requirements. Smaller models (7B-8B params) run well on a single consumer GPU (RTX 3090/4090, 24 GB VRAM). Larger models (30B-70B) benefit from professional GPUs (A100, H100) or multiple consumer GPUs. With quantisation and KV cache compression, xinfer can run 30B+ models on single 24-48 GB GPUs. We assess your workload and recommend the minimum viable hardware — often starting with a cloud GPU instance before committing to on-prem hardware.
How much does it cost compared to API pricing?
At low volume (<1M tokens/month), API pricing is cheaper because there is no fixed infrastructure cost. At high volume (10M+ tokens/month), self-hosted typically breaks even within 3-6 months. After break-even, the marginal cost of each additional request is near zero — just electricity and GPU wear. We build a cost model for your specific workload during the assessment phase so you know the numbers before committing.
Is it hard to set up and maintain?
Candle and xinfer are designed to minimise operational overhead — they ship as single static binaries with no Python, Node, or Docker dependencies. Deploying the initial setup typically takes 1-2 days. Ongoing maintenance involves model updates (a few times per year) and hardware monitoring. We handle the initial deployment and can provide ongoing managed services if your team prefers not to operate the infrastructure directly.
Can I use a hybrid approach — API for some workloads, self-hosted for others?
Absolutely. This is the recommended architecture. Use frontier API models for complex reasoning, creative tasks, and anything requiring the latest capabilities. Route high-volume, sensitive, or latency-critical tasks to your self-hosted inference server. Both Candle and xinfer expose OpenAI-compatible APIs, so your application code treats both the same way — just point at a different endpoint. We help you design and implement this routing.
